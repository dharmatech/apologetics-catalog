# v0.1 YAML Authoring Schema

## Purpose

This document defines the human-readable authoring contract for
`schema_version: "0.1"` YAML files.

It describes author-authored YAML document envelopes, entity records, and
embedded objects.

It does not fully specify the normalized catalog JSON, graph projection JSON, or
other generated outputs.

The v0.1 schema should keep required fields minimal so the first dataset can be
modeled without unnecessary friction.

Controlled field values are defined separately in:

```text
/docs/design/10-vocabularies.md
```

---

## Document Envelope

Every YAML document requires:

```yaml
schema_version: "0.1"

kind: topic
```

Required fields:

```text
schema_version
kind
```

Notes:

* `schema_version` must be a quoted string.
* `kind` identifies the document shape, not necessarily one entity kind.
* Topic-centered bundle documents may contain multiple record sections.
* Single-record documents should use the same envelope style.

Common document kinds:

```text
project
topic
question
claim
tradition
agent
position
source
evidence
interpretation
assumption
argument
relationship
```

---

## Field Conventions

Single references use `_id`.

```yaml
question_id: question.christology.created_being
```

Reference lists use `_ids`.

```yaml
related_question_ids:
  - question.christology.firstborn_meaning
```

Structured references may use an object when both a target kind and target ID
are needed.

```yaml
holder:
  type: tradition
  id: tradition.jehovahs_witnesses
```

Sections containing multiple records should be lists of records.

```yaml
claims:
  - id: claim.jesus_created
    question_id: question.christology.created_being
    summary: >
      Jesus is a created being.
```

Every entity record requires an explicit stable `id`.

## Project

Purpose:

Project-level metadata and configuration.

Required fields:

```text
id
title
```

Optional fields:

```text
description
vocabulary_extensions
```

Notes:

`vocabulary_extensions` may also appear at the document level in a project
document.

---

## Topic

Purpose:

Lightweight classification for questions.

Required fields:

```text
id
title
```

Optional fields:

```text
description
```

Notes:

Topics classify questions. They are not the primary debate container.

---

## Question

Purpose:

Central issue or debate container.

Required fields:

```text
id
title
```

Optional fields:

```text
question_type
status
topic_ids
parent_question_id
description
```

Reference fields:

```text
topic_ids
parent_question_id
```

Controlled fields:

```text
question_type
status
```

Notes:

Use `parent_question_id` for one primary hierarchy. Use first-class
relationships for non-hierarchical question links.

---

## Claim

Purpose:

Atomic proposition related to a question.

Required fields:

```text
id
question_id
summary
```

Optional fields:

```text
proposition
scope
qualifiers
related_question_ids
provenance
```

Reference fields:

```text
question_id
related_question_ids
```

Notes:

Opposing claims should be separate records connected by first-class
relationships. Do not use a claim `polarity` field in v0.1.

---

## Tradition

Purpose:

Group, school, denomination, movement, or intellectual tradition.

Required fields:

```text
id
name
```

Optional fields:

```text
description
aliases
```

---

## Agent

Purpose:

Entity capable of making, holding, preserving, transmitting, or being attributed
with claims, arguments, interpretations, or positions.

Required fields:

```text
id
type
name
```

Optional fields:

```text
description
aliases
```

Controlled fields:

```text
type
```

Notes:

Publications are usually `Source` records, not `Agent` records.

---

## Position

Purpose:

Stance attributed to one holder in relation to one or more claims.

Required fields:

```text
id
holder
claim_stances
```

Optional fields:

```text
summary
provenance
```

Embedded objects:

```text
holder
claim_stance
```

Notes:

Use one holder per position in v0.1. Use separate positions when multiple
holders share a stance.

---

## Source

Purpose:

Work, edition, translation, manuscript, publication, article, dataset, or other
source of evidence.

Required fields:

```text
id
type
title
```

Optional fields:

```text
work_id
rights
author
date
url
language
edition
description
```

Reference fields:

```text
work_id
```

Controlled fields:

```text
type
```

Embedded objects:

```text
rights
```

Notes:

Exact quotations should normally point to the concrete source whose wording is
quoted, not to an abstract work.

---

## Evidence

Purpose:

Extracted material from a concrete source at a locator.

Required fields:

```text
id
source_id
locator
```

At least one content field should be present:

```text
quotation
paraphrase
data
```

Optional fields:

```text
quotation
paraphrase
data
provenance
```

Reference fields:

```text
source_id
```

Embedded objects:

```text
locator
provenance
```

Notes:

Evidence should contain specific material, not merely a citation.

---

## Interpretation

Purpose:

Reading of evidence.

Required fields:

```text
id
evidence_id
summary
```

Optional fields:

```text
related_evidence_ids
method
provenance
```

Reference fields:

```text
evidence_id
related_evidence_ids
```

Controlled fields:

```text
method
```

Notes:

An interpretation says what evidence is taken to mean. Arguments perform the
inferential move from interpretations toward claims.

---

## Assumption

Purpose:

Reusable premise required by an interpretation or argument.

Required fields:

```text
id
summary
```

Optional fields:

```text
category
provenance
```

Controlled fields:

```text
category
```

Notes:

Dependencies on assumptions should be represented through first-class
relationships.

---

## Argument

Purpose:

Reasoned move that supports, challenges, qualifies, clarifies, or responds to
another entity.

Required fields:

```text
id
role
summary
```

Optional fields:

```text
provenance
```

Controlled fields:

```text
role
```

Notes:

Arguments should target or use other entities through first-class relationships,
not by embedding target lists directly.

---

## Relationship

Purpose:

First-class typed connection between two entities.

Required fields:

```text
id
type
from_id
to_id
```

Optional fields:

```text
summary
provenance
```

Reference fields:

```text
from_id
to_id
```

Controlled fields:

```text
type
```

Notes:

Use first-class relationships for argumentative, interpretive, claim-to-claim,
question-to-question, dependency, response, and other metadata-bearing links.

---

## Embedded Object: Locator

Purpose:

Location within a source.

Required fields:

```text
type
value
```

Optional fields:

```text
label
```

Controlled fields:

```text
type
```

---

## Embedded Object: Provenance

Purpose:

Attribution for a derived statement or relationship.

Common optional fields:

```text
attributed_to
work
date
locator
url
accessed
edition
language
```

Embedded objects:

```text
locator
```

Notes:

No single provenance field is globally required in v0.1, but provenance should
be structured when present.

---

## Embedded Object: Rights

Purpose:

Rights and licensing metadata for a source.

Optional fields:

```text
status
license
attribution
url
```

Controlled fields:

```text
status
```

Notes:

Generated outputs may later use rights metadata for attribution or quotation
rules.

---

## Embedded Object: Holder

Purpose:

Typed reference to the holder of a position.

Required fields:

```text
type
id
```

Controlled fields:

```text
type
```

Notes:

The holder `type` should identify the kind of holder being referenced, such as
`tradition`, `person`, `organization`, or `council`.

---

## Embedded Object: Claim Stance

Purpose:

Position holder's stance toward a claim.

Required fields:

```text
claim_id
stance
status
```

Reference fields:

```text
claim_id
```

Controlled fields:

```text
stance
status
```

---

## Embedded Object: Source Ref

Purpose:

Compiler metadata identifying where a record came from.

Fields:

```text
file
line
column
```

Notes:

`source_ref` is normally generated by the compiler for diagnostics and the
normalized catalog. Authors should not need to write it in source YAML.
